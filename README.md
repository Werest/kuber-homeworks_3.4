### Задание 1. Выбрать стратегию обновления приложения и описать ваш выбор

1. Имеется приложение, состоящее из нескольких реплик, которое требуется обновить.
2. Ресурсы, выделенные для приложения, ограничены, и нет возможности их увеличить.
3. Запас по ресурсам в менее загруженный момент времени составляет 20%.
4. Обновление мажорное, новые версии приложения не умеют работать со старыми.
5. Вам нужно объяснить свой выбор стратегии обновления приложения.
------------------------------------------
- **RollingUpdate**: Не подходит из-за отсутствия обратной совместимости. Поды v1 и v2 будут смотреть в одну БД/использовать общие протоколы, что вызовет сбои.
- **Recreate (Полная остановка -> Обновление -> Запуск)**: Вызывает downtime, неприемлемо для современных приложений.
- **Blue-Green**: Требует 200% ресурсов (запуск двух полных сред), что запрещено условием.
- **A/B-тестирование**: Похоже на Canary, но обычно подразумевает более долгий параллельный запуск версий для фич-тестирования. Наш кейс - после проверки функционала нам нужно полностью перейти на v2.

Наш выбор: Canary-развертывание с контролируемым переключением трафика (Canary Release)

**Причины:**
- Обратной совместимости нет -> Rolling Update не подходит
- Ресурсы ограничены -> Blue-Green не подходит (нужно 200% ресурсов)
- Есть 20% запаса -> Можно запустить 1-2 пода новой версии для тестирования
- Безопасность и контроль -> Можно тестировать на части трафика и быстро откатиться

**Как работает:**
- Запускаем новую версию рядом со старой (в пределах 20% ресурсов)
- Направляем на неё сначала тестовый, затем небольшой % продакшн-трафика
- После проверки постепенно увеличиваем трафик до 100%
- Удаляем старую версию

**Canary можно представить в следующем виде:**
```
Этап 1: [v1][v1][v1] - 100% трафика
Этап 2: [v1][v1][v1] - 95% трафика
        [v2] - 5% трафика
Этап 3: [v1][v1] - 50% трафика
        [v2][v2] - 50% трафика
Этап 4: [v2][v2][v2] - 100% трафика
```
